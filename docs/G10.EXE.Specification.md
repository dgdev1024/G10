# Specification: G10 Executable File Format

Below is the detailed specification for the G10 Executable File Format (`.g10`).
This is the final, link-time image produced by the G10 Linker. A conforming
G10 runtime or loader should be able to load and run a `.g10` file produced by
the assembler toolchain.

## Overview

A G10 executable file contains layout information and the raw bytes for each
program section in a compact, relocations-resolved form. The linker combines
multiple object files (`.o`) into one `.g10` executable: it assigns final
addresses to sections, applies relocations, and produces a symbol table with
resolved addresses.

The executable format intentionally omits the string table and relocation
records found in object files — symbol names and relocations are resolved at
link time. Optional debug or metadata may be carried inside dedicated
`Metadata` sections.

## Endianness and Data Types

- All multi-byte integer fields (16-bit, 32-bit) are encoded in **Little-Endian**
  format unless otherwise noted.
- The file's 4-byte magic number is stored **Big-Endian** (this aids easy file
  identification on both little and big endian hosts).
- Offsets and sizes are 32-bit unsigned integers. Counts are 32-bit unsigned
  integers. Addresses are 32-bit unsigned values representing G10's 32-bit
  address space.

Constants used by readers and writers:

- Magic: ASCII "G10X" — 0x47313058 (4 bytes, Big-Endian)
- Version: Encoded as (Major * 10000) + (Minor * 100) + Patch (32-bit LE)

## Overall File Structure

The file is divided into contiguous regions. All offsets present in header
structures are file offsets measured from the start of the file (byte 0).

1. Program Header
2. Section Table
3. Symbol Table
4. Section Data Buffer

Readers must use offsets from the header to locate and validate subsequent
tables and the data buffer. The canonical serialization order recommended for
writers is: write placeholder header, write section table, write symbol table,
write data buffer, then backfill header with computed offsets and counts.

## 1. Program Header

The header is 32 bytes in total and appears at file offset 0.

| Offset  | Size (Bytes) | Field Name              | Description |
|---------|--------------|-------------------------|-------------|
| `$0000` | 4            | Magic Number            | Must be 0x47313058 ("G10X"), stored Big-Endian. |
| `$0004` | 4            | Format Version          | Encoded version (Major * 10000 + Minor * 100 + Patch), 32-bit LE. |
| `$0008` | 4            | Flags                   | `ProgramFlags` (32-bit LE). Reserved; value 0 for now. |
| `$000C` | 4            | Section Table Offset    | File offset to the Section Table (32-bit LE). |
| `$0010` | 4            | Section Count           | Number of entries in the Section Table (32-bit LE). |
| `$0014` | 4            | Symbol Table Offset     | File offset to the Symbol Table (32-bit LE). |
| `$0018` | 4            | Symbol Count            | Number of entries in the Symbol Table (32-bit LE). |
| `$001C` | 4            | Data Buffer Offset      | File offset to the Section Data Buffer (32-bit LE). |

Notes:

- The header must be the first bytes of the file so a loader can quickly
  validate the magic/version before reading other regions.
- Readers should validate that all offsets + sizes read from the tables are
  within the file bounds.

## 2. Section Table

The Section Table is an array of fixed-size entries describing each section
contained by the program. Each entry is 16 bytes.

ProgramSectionEntry layout (16 bytes):

| Offset | Size | Field | Encoding |
|--------|------|-------|----------|
| 0x00   | 1    | mType | `ProgramSectionType` (byte) |
| 0x01   | 1    | mInterruptNumber | byte (valid for `Interrupt` sections) |
| 0x02   | 2    | mAlignmentBoundary | 16-bit LE — alignment in bytes (power-of-two boundary recommended) |
| 0x04   | 4    | mTargetAddress | 32-bit LE — final load address for the section in memory |
| 0x08   | 4    | mDataSize | 32-bit LE — size in bytes of the section's data payload |
| 0x0C   | 4    | mDataOffset | 32-bit LE — offset into the Data Buffer where the section's data begins |

Semantics:

- `mType` — one of Metadata, Interrupt, Code, Data, BSS.
- `mTargetAddress` — the absolute address where the section will be placed at
  runtime.
- `mDataSize` & `mDataOffset` — together indicate a contiguous slice inside
  the Data Buffer. The physical location of the section's bytes in the file
  is: `DataBufferStart + mDataOffset`, with length `mDataSize`.
- `BSS` sections reserve memory but do not occupy bytes in the Data Buffer.
  For `BSS`, `mDataSize` must be set to the number of zero-initialized bytes
  required; `mDataOffset` is typically 0 or ignored — readers must allocate a
  zero-filled region of `mDataSize` at `mTargetAddress`.

Alignment:

- Section writers should place section data within the Data Buffer respecting
  `mAlignmentBoundary` where possible. Aligning `mDataOffset` to the
  `mAlignmentBoundary` value (or to 4 bytes if unspecified) simplifies loaders.

Ordering:

- The linker may choose any order for entries in the Section Table. A common
  practice is to list sections in the order they appear in the Section Data
  Buffer, but this is not required so long as offsets are correct.

## 3. Symbol Table

The Symbol Table is an array of fixed-size entries. Each entry is 8 bytes.

ProgramSymbolEntry layout (8 bytes):

| Offset | Size | Field | Encoding |
|--------|-----:|-------|----------|
| 0x00   | 4    | mAddressOffset | 32-bit LE — offset relative to the section's `mTargetAddress` |
| 0x04   | 4    | mSectionIndex   | 32-bit LE — index into the Section Table identifying the symbol's section |

Semantics:

- The symbol's absolute address is computed as:

  `symbol_address = SectionTable[mSectionIndex].mTargetAddress + mAddressOffset`

- Only numeric symbol references are stored (address + section index); the
  executable image does not carry the symbol name string table. If link-time
  symbol names are required for debugging, the linker can place that
  information inside a `Metadata` section using a project-specific encoding.

Validation:

- `mSectionIndex` must be less than `Section Count`.
- `mAddressOffset + symbol_size` (where applicable) must be within the
  corresponding section's `mDataSize` (for non-BSS sections) or within the
  allocated BSS range for BSS symbols.

## 4. Section Data Buffer

The Data Buffer is a contiguous region of bytes containing the payload for
all non-BSS sections. The start of the buffer is given by the header field
`DataBufferOffset`. Each section entry's `mDataOffset` indexes into this
buffer to the start of that section's bytes.

Rules:

- A section described as non-BSS must have its `mDataOffset` and `mDataSize`
  set so that the slice lies entirely inside the Data Buffer.
- The linker may pack multiple section data blocks back-to-back. Gaps are
  permitted (and may be used to satisfy alignments), but readers must rely on
  `mDataOffset`/`mDataSize` rather than assume contiguous ordering.

BSS sections should not consume bytes in the Data Buffer; instead the
loader should allocate and zero memory for the `mDataSize` bytes specified by
the section entry at the `mTargetAddress`.

## Linking and Resolution

When producing an executable the linker performs these high-level steps:

1. Decide final `mTargetAddress` for each section and fill the Section Table.
2. Lay out non-BSS section payloads into the Data Buffer and set `mDataOffset`
   and `mDataSize` for each section.
3. Apply all relocations so code/data are fixed up to their final addresses.
4. Build the Symbol Table with `mAddressOffset` values computed relative to
   each section's `mTargetAddress`.
5. Write tables and data to disk, backfilling header offsets and counts.

After step 3 the executable contains no unresolved relocations — all addresses
are final.

## Serialization / Reader Guidance

Recommended reading order (safe):

1. Read the first 4 bytes and verify the magic number is 0x47313058 (Big-Endian).
2. Read the remaining header fields (32-bit LE fields) and verify offsets/counts
   are within the file size.
3. Seek to `SectionTableOffset` and read `SectionCount` entries (each 16 bytes).
4. Seek to `SymbolTableOffset` and read `SymbolCount` entries (each 8 bytes).
5. Seek to `DataBufferOffset` and validate that the sum of each section's
   `mDataOffset + mDataSize` is within the buffer.
6. For each section, load the data from `DataBufferOffset + mDataOffset`.
   For BSS sections, allocate zero-initialized memory of `mDataSize` at
   `mTargetAddress`.

Security / robustness checks:

- Verify that `SectionTableOffset`, `SymbolTableOffset`, and `DataBufferOffset`
  are greater than or equal to the header size and do not overlap in ways that
  violate the file size.
- Validate that indices (e.g., symbol `mSectionIndex`) are within bounds.
- Check for arithmetic overflow when adding offsets + sizes.

## Versioning and Compatibility

- The `Format Version` field allows readers to detect incompatible file format
  changes. If an incompatible version is encountered, the reader should reject
  the file with a clear diagnostic message.
- The `Flags` field in the header is reserved for future uses.

## Example — Minimal Layout

Assume a tiny program with one `Code` section of 16 bytes placed at
target address 0x00100000 and one symbol at offset 0x0 inside that section.

- Header (32 bytes)
- Section Table (1 entry × 16 bytes = 16 bytes)
- Symbol Table (1 entry × 8 bytes = 8 bytes)
- Data Buffer (16 bytes)

Header values (conceptual):

- `SectionTableOffset` = 0x20 (immediately after header)
- `SectionCount` = 1
- `SymbolTableOffset` = 0x20 + 0x10 = 0x30
- `SymbolCount` = 1
- `DataBufferOffset` = 0x30 + 0x08 = 0x38

On disk the file looks like:

| Offset | Contents |
|--------|----------|
| 0x00   | Program Header (32 bytes) |
| 0x20   | Section Entry (16 bytes) |
| 0x30   | Symbol Entry (8 bytes) |
| 0x38   | Section Data (16 bytes) |

## Remarks

- The executable format intentionally keeps symbol names out of the core
  image to keep runtime images compact. Tooling that requires symbol names
  should store them in a `Metadata` section or in separate debug files.
- This specification is aligned with the G10 object file format but
  intentionally drops relocation and string-table structures which are only
  meaningful during link time.

See also: the G10 Object File Format specification for details on the
input format that the linker consumes when producing this executable.
