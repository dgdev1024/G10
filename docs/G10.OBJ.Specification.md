# Specification: G10 Object File Format

Below is the detailed specification for the G10 Object File Format (`.o`), detailing the structure, headers, and tables generated by the G10 CPU Assembler during the Code Generation phase, which are subsequently processed by the Linker.

## Overview

The G10 assembler produces relocatable object files (`.o` extension) during the assembly of a source module. These object files contain compiled machine code, data definitions, and metadata (such as symbols and relocations) required by the linker to combine multiple object files into a single, cohesive executable format (`.g10`).

### Endianness and Data Types

As dictated by the G10 CPU architecture, all metadata fields, offsets, counts, and addresses within the object file format are stored in **Little-Endian** format (least significant byte first), unless explicitly representing a CPU instruction opcode, which the G10 strictly evaluates as Big-Endian.

Offsets and memory addresses are universally represented as unsigned 32-bit integers, reflecting the G10's 32-bit address space.

## Overall Structure

A G10 object file consists of the following contiguous sections:

1. **Header**: Magic number, version, and metadata offsets.
2. **Section Table**: List of sections generated by the assembler with their attributes and locations.
3. **Symbol Table**: Declarations for exported, imported, global, and local symbols.
4. **Relocation Table**: Address references requiring linker fixups.
5. **String Table**: A null-terminated string repository for symbol and section names.
6. **Section Data**: The raw, assembled binary code and data.

---

## 1. Header

The object file begins with a 32-byte header providing essential metadata and pointers to the rest of the file's structures.

| Offset | Size (Bytes) | Field Name              | Description |
|--------|--------------|-------------------------|-------------|
| `$0000`  | 4            | **Magic Number** | Identifies the file type. Must be `"G10O"` (`$4731304F`). |
| `$0004`  | 4            | **Format Version** | Object file format version (e.g. Major * 10000 + Minor * 100 + Patch) |
| `$0008`  | 4            | **Flags** | File-wide flags (currently unused). |
| `$000C`  | 4            | **Section Table Offset**| Byte offset from the start of the file to the Section Table. |
| `$0010`  | 4            | **Section Count** | Number of entries in the Section Table. |
| `$0014`  | 4            | **Symbol Table Offset** | Byte offset from the start of the file to the Symbol Table. |
| `$0018`  | 4            | **Symbol Count** | Number of entries in the Symbol Table. |
| `$001C`  | 4            | **Reloc Table Offset** | Byte offset from the start of the file to the Relocation Table. |
| `$0020`  | 4            | **Reloc Count** | Number of entries in the Relocation Table. |
| `$0024`  | 4            | **String Table Offset** | Byte offset from the start of the file to the String Table. |
| `$0028`  | 4            | **String Buffer Size** | The size, in bytes, of the String Table's buffer. |
| `$002C`  | 4            | **String Count** | Number of strings in the String Table, including the index zero null string. |
| `$0030`  | 4            | **Data Buffer Offset** | Byte offset from the start of the file to the Section Data Buffer. |

---

## 2. Section Table

The Section Table describes the various memory sections (`METADATA`, `CODE`, `DATA`, `BSS`, `INT0-INT31`) declared in the assembly source. Each entry defines the section's properties, constraints, and points to its raw data payload. 

*Note: Since the base G10 specification specifies a section table but does not detail its exact byte-level format, the following 24-byte entry structure is standardized for the G10 ecosystem.*

### Section Entry Format (24 bytes per entry)

| Offset | Size (Bytes) | Field Name              | Description |
|--------|--------------|-------------------------|-------------|
| `$0000`  | 4            | **Name Offset** | Byte offset into the String Table for the section's name. |
| `$0004`  | 1            | **Section Type** | Evaluated type of the section (see Section Types below). |
| `$0005`  | 1            | **Interrupt Number** | If the section is an interrupt vector, this specifies its number (0-31). |
| `$0006`  | 2            | **Alignment** | Byte boundary alignment requirement (e.g., 4, 16, 256). |
| `$0008`  | 4            | **Target Address** | Optional fixed origin address (`.ORG`). `$FFFFFFFF` if relocatable. |
| `$000C`  | 4            | **Data Size** | Size of the section data in bytes. |
| `$0010`  | 4            | **Data Offset** | Byte offset in the object file where the raw section data begins. |

### Section Types

- `$00`: `METADATA` (Restricted to `$00000000` - `$00000FFF`)
- `$01`: `INT0` - `INT31` (Fixed vector spaces in `$00001000` - `$00001FFF`)
- `$02`: `CODE` / `TEXT` (Executable region)
- `$03`: `DATA` / `RODATA` (Read-only data region)
- `$04`: `BSS` (Uninitialized RAM - Data Offset is ignored, Data Size indicates reservation)

---

## 3. Symbol Table

The Symbol Table acts as the directory for all named labels, variables, and imported/exported references. The linker uses these to resolve dependencies across multiple `.o` files.

### Symbol Entry Format (12 bytes per entry)

| Offset | Size (Bytes) | Field Name              | Description |
|--------|--------------|-------------------------|-------------|
| `$0000`  | 4            | **Name Offset** | Byte offset into the String Table for the symbol's identifier. |
| `$0004`  | 2            | **Symbol Type** | Defines visibility and usage (see below). |
| `$0006`  | 4            | **Address Offset** | The address offset within its section. |
| `$000A`  | 4            | **Section Index** | 0-based index into the Section Table, or `$FFFFFFFF` if absolute/external. |

### Symbol Types

- `$0000`: Local (Scoped to this object file; usually stripped in release builds)
- `$0001`: Export (Defined here, can be referenced by other modules via `.EXPORT` / `.GLOBAL`)
- `$0002`: Import (Defined externally, required by this module via `.IMPORT` / `.EXTERN`)

---

## 4. Relocation Table

Relocations are placeholders left by the assembler when an instruction or data directive references a label whose final address is unknown until link-time. The linker patches the raw section data using these entries once final memory layout is determined.

### Relocation Entry Format (12 bytes per entry)

| Offset | Size (Bytes) | Field Name              | Description |
|--------|--------------|-------------------------|-------------|
| `$0000`  | 4            | **Patch Offset** | Byte offset within the target section where the patch must be applied. |
| `$0004`  | 4            | **Symbol Index** | 0-based index into the Symbol Table representing the target value. |
| `$0008`  | 4            | **Section Index** | 0-based index into the Section Table denoting which section requires the fixup. |
| `$000C`  | 1            | **Relocation Type** | The math applied during the patch (see below). |
| `$000D`  | 1            | **Size** | Size of the value to patch (1, 2, or 4 bytes). |

### Relocation Types

- `$00`: **Absolute Address** (Used by `JMP IMM32`, `CALL IMM32`, `LD DX, [label]`, `.DD label`). The linker simply overwrites the patch location with the target label's address, truncated according to the relocation's size if necessary.
- `$01`: **Relative Address** (Used by `JPB SIMM16`). The linker calculates the distance between the current Instruction Pointer (`PC`) and the target symbol, writing the signed difference.

---

## 5. String Table

To avoid wasting space with fixed-length string fields, all variable-length names (section names, symbol names) are stored in the String Table. 

The String Table immediately follows the Relocation Table in the file (its exact offset can be calculated as `Reloc Table Offset + (Reloc Count * 12)`).

- It is a contiguous block of bytes containing ASCII/UTF-8 strings.
- Every string is null-terminated (`\0`).
- The very first byte (offset 0) is a null byte, allowing a `Name Offset` of `0` to safely represent an anonymous/unnamed entity.

---

## 6. Section Data

The raw binary payload of the object file. This region stores the compiled G10 instructions (big-endian opcodes, little-endian operands) and structured data directives (`.DB`, `.DW`, `.DD`, `.DZ`). 

- The location and size of each data block are dictated by the corresponding `Data Offset` and `Data Size` fields in the Section Table.
- `BSS` sections do not occupy space in the Section Data block, as they represent uninitialized memory. Their `Data Size` simply informs the linker how much RAM to reserve at `$80000000+`.

---

## Example Hex Dump Walkthrough

If the assembler outputs a file for a `main.asm` containing a simple `JMP start`, the object file might logically group as:

1. **`$0000`**: `47 31 30 4F` (`"G10O"`)
2. **`$0004`**: `01 00` (v1.0)
3. **`...`**: Pointers to the subsequent tables
4. **Section Table**: Entry for `.SECTION "Main", CODE`
5. **Symbol Table**: Entry for `start`
6. **Reloc Table**: Entry telling the linker to patch the 4-byte immediate following the `JMP` opcode at offset `$0002` in the `CODE` section.
7. **String Table**: `\0Main\0start\0`
8. **Section Data**: `40 00 00 00 00 00` (`JMP NC, $00000000` — zeroes acting as a placeholder until the linker applies Relocation `$00`).
